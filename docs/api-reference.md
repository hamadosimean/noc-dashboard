## Authentication

Two independent auth schemes are used, depending on the caller:

| Scheme | Header | Used by | Endpoints |
|---|---|---|---|
| **Static API key** | `Authorization: Bearer $NOC_API_KEY` | Supervision-tool webhooks / the ETL collectors | `POST /api/incidents/ingest`; also accepted (alternative to a JWT) on `GET /api/report/monthly` for the scheduled ETL export |
| **JWT bearer** | `Authorization: Bearer <access_token>` | The dashboard frontend, after login | Every `GET` KPI/SLA/alerts/locality/report endpoint, `PATCH /api/incidents/{id}/acknowledge`, `PATCH /api/incidents/{id}/resolve`, `GET /api/auth/me`, and the `/ws/alerts` WebSocket (token as query param) |

- JWTs are issued by `POST /api/auth/login` or `POST /api/auth/pin-login`, signed
  with `HS256` using `SECRET_KEY`, and expire after `ACCESS_TOKEN_EXPIRE_MINUTES`
  (default 30). Payload: `{sub, username, role, exp}`.
- **Every endpoint requires authentication** (spec §10.1). Reads accept any
  logged-in role; write actions are role-gated by `require_role()`
  (`backend/app/core/security.py`):

| Role | Reads | `acknowledge` | `resolve` |
|---|---|---|---|
| `admin` | ✅ | ✅ | ✅ |
| `noc_agent` | ✅ | ✅ | ✅ |
| `analyst` | ✅ | ❌ `403` | ❌ `403` |

## Rate limiting

Per-IP, per-minute fixed windows backed by Redis
(`backend/app/core/rate_limit.py`, spec §10.1). Behind NGINX, the client IP is
taken from `X-Real-IP`.

| Scope | Limit | Env var |
|---|---|---|
| All read endpoints (KPI/SLA/alerts/locality/report, shared budget) | 100 req/min | `RATE_LIMIT_READ_PER_MIN` |
| `POST /api/incidents/ingest` | 10 req/min | `RATE_LIMIT_INGEST_PER_MIN` |

Exceeding a limit returns `429` with a `Retry-After: 60` header. Limiting is
**fail-open** (a Redis outage lets requests through) and can be disabled
entirely with `RATE_LIMIT_ENABLED=false`.

## Conventions

- Nearly every KPI/SLA endpoint takes `month` (1–12) and `year` as required
  query params and returns data scoped to that calendar month.
- Responses are cached in Redis per the rules in
  [architecture.md#caching-strategy](architecture.md#caching-strategy); a
  cache hit returns byte-identical JSON to the original computation.
- Timestamps are ISO 8601. `fact_incident` columns are stored as naive UTC;
  the backend normalizes any timezone-aware input before writing.

## KPI endpoints

All under `/api/kpi` unless noted.

### `GET /api/kpi/summary`

Monthly KPI headline numbers plus delta vs. the previous month.

Query: `month`, `year`

```json
{
  "period": { "month": 7, "year": 2026, "label": "Juillet 2026" },
  "kpi": {
    "total_incidents": 142,
    "resolved": 128,
    "open": 14,
    "resolution_rate_pct": 90.1,
    "avg_mttr_minutes": 47.3,
    "network_availability_pct": 97.8,
    "critical_localities": 2,
    "recurrent_nodes": 6,
    "off_hours_detected": 19
  },
  "vs_previous_month": { "incidents_delta": -8, "availability_delta": 0.4 }
}
```

`critical_localities` = localities with `availability_pct < 95.0` this month.
`recurrent_nodes` = nodes with ≥ 3 incidents this month. `off_hours_detected`
= incidents whose generated `shift` column is `'auto'` (outside the 06:00–22:00
NOC / 07:00–17:00 terrain windows).

### `GET /api/kpi/compare`

Current month vs earlier periods (N-1 and N-3 months), with per-KPI deltas —
the spec's P2 "Comparaison périodes" module. Feeds the "Comparaison de
Périodes" card on the Vue Globale tab.

Query: `month`, `year`

```json
{
  "period": { "month": 7, "year": 2026, "label": "Juillet 2026" },
  "kpi": { "total_incidents": 142, "...": "..." },
  "comparisons": [
    {
      "offset_months": 1,
      "period": { "month": 6, "year": 2026, "label": "Juin 2026" },
      "kpi": { "total_incidents": 150, "...": "..." },
      "deltas": {
        "total_incidents": -8,
        "resolved": 4,
        "resolution_rate_pct": 2.1,
        "avg_mttr_minutes": -5.0,
        "network_availability_pct": 0.4
      }
    },
    { "offset_months": 3, "...": "..." }
  ]
}
```

`deltas` are always `current − past`, so a negative `total_incidents` delta
means fewer incidents than the comparison month.

### `GET /api/kpi/localities`

Top localities by incident count, for months where they had activity.

Query: `month`, `year`, `limit` (default 10, max 100)

Returns a list of `{locality_id, locality, region, latitude, longitude,
total_incidents, resolved, avg_mttr, availability_pct}`.

### `GET /api/kpi/localities/map`

Same shape as above, but returns **every** locality with coordinates —
including ones with zero incidents this month — so the supervision map never
has a "missing" node. Feeds `BurkinaFasoMap.jsx`.

### `GET /api/kpi/nodes`

Top nodes by incident count, optionally scoped to a locality.

Query: `month`, `year`, `locality_id` (optional), `limit` (default 10, max 100)

Returns `{node_id, code, name, locality, source_tool, total_incidents,
resolved, avg_mttr, availability_pct}[]`.

### `GET /api/kpi/recurrent`

Nodes with incident counts at or above a threshold this month — the "repeat
offenders" list.

Query: `month`, `year`, `min_count` (default 3)

Returns `{node_id, code, name, locality, total_incidents}[]`.

### `GET /api/kpi/trend`

Last N months of incidents/availability, oldest first.

Query: `months` (default 6, max 24), `year`/`month` (optional — defaults to
the latest month with data in `mv_kpi_node_monthly`)

Returns `{month, year, label, total_incidents, resolved, avg_mttr,
availability_pct}[]`.

### `GET /api/kpi/hour-distribution`

Incident counts bucketed by hour of day (0–23, all 24 always present, zero-filled).

Query: `month`, `year`

### `GET /api/kpi/causes`

Incident counts and average MTTR grouped by cause category (e.g. Énergie,
Équipement, Liaison, Logiciel, Humain).

Query: `month`, `year`

### `GET /api/locality/{locality_id}/nodes`

Node-level detail for one locality (used by the "Vue par Localité" tab).

Query: `month`, `year`
Path: `locality_id`

Returns `404` if the locality doesn't exist. Otherwise:

```json
{
  "locality_id": 4,
  "locality": "Ouagadougou",
  "region": "Centre",
  "period": { "month": 7, "year": 2026, "label": "Juillet 2026" },
  "nodes": [
    {
      "node_id": 12, "code": "OUA-012", "name": "...", "node_type": "router",
      "source_tool": "zabbix", "is_active": true,
      "total_incidents": 5, "resolved": 4, "open": 1,
      "avg_mttr": 32.1, "availability_pct": 98.4
    }
  ]
}
```

## SLA endpoint

### `GET /api/sla`

SLA indicators vs. their targets, derived from the same month's KPI summary.

Query: `month`, `year`

```json
{
  "period": { "month": 7, "year": 2026, "label": "Juillet 2026" },
  "indicators": [
    { "metric": "Disponibilité Cœur de Réseau", "value": 99.6, "target": 99.5, "status": "met" },
    { "metric": "Disponibilité Nœuds d'Accès", "value": 97.8, "target": 95.0, "status": "met" },
    { "metric": "Taux de Résolution < 4h", "value": 90.1, "target": 80.0, "status": "met" }
  ]
}
```

`status` is `"met"` if `value >= target`, else `"not_met"`. "Core" availability
is averaged only across `centreon`/`netxms`-sourced nodes (the tools that
front core network equipment in this deployment).

## Alerts endpoint

### `GET /api/alerts/open`

Open or acknowledged alerts, oldest-first (i.e. the ones that have been open
longest surface first) — **not cached**, to stay near-real-time.

Query: `limit` (default 20, max 100)

Returns `{id, node_code, node_name, locality, severity, status, description,
detected_at, age_minutes}[]`.

## Incidents endpoints

### `POST /api/incidents/ingest`

Webhook entry point for supervision tools (and the ETL collectors). Requires
the static API key header.

**Auth**: `Authorization: Bearer $NOC_API_KEY`

Request body:

```json
{
  "external_id": "zabbix-alert-483920",
  "source_tool": "zabbix",
  "node_code": "OUA-012",
  "severity": "high",
  "status": "open",
  "detected_at": "2026-07-07T09:14:00Z",
  "description": "Batterie onduleur déchargée",
  "cause_category": "Énergie",
  "cause_label": "Batterie onduleur déchargée",
  "itop_auto_ticket": true
}
```

`source_tool` must be one of `zabbix|nagios|netxms|centreon`. `severity` one
of `critical|high|medium|low` (default `medium`). `status` one of
`open|acknowledged|resolved|closed` (default `open`). `node_code` must match
an existing `dim_node.code` or the request 404s.

Response (`201 Created`):

```json
{ "incident_id": 5821, "node_id": 12, "itop_ticket_id": "TKT-2026-05821", "shift": "noc", "created_at": "2026-07-07T09:14:03" }
```

Side effects: creates the cause dimension row if new, optionally mints an
iTop ticket reference, refreshes `mv_kpi_node_monthly` (when
`SYNC_MV_REFRESH=true`), invalidates all `kpi:*` cache keys, **publishes the
incident to the `/ws/alerts` WebSocket stream** (via Redis pub/sub), and — for
`critical` severity with `NOTIFICATIONS_ENABLED=true` — fires SMS (Twilio) +
email (SMTP) to the NOC as a background task
(`backend/app/services/notification_service.py`). Notification failures are
logged, never surfaced to the webhook caller.

### `PATCH /api/incidents/{incident_id}/resolve`

**Auth**: JWT bearer — roles `admin` or `noc_agent` (`403` for `analyst`)

Body: `{ "resolved_at": "2026-07-07T10:00:00Z", "notes": "Onduleur remplacé" }`
(both optional — `resolved_at` defaults to now).

Marks the incident `resolved`, computes `downtime_minutes`, back-fills
`acknowledged_at` if it was never set, appends `notes` to the description,
refreshes the KPI view, and invalidates the cache. Returns the same shape as
`ingest`. `404` if the incident doesn't exist.

### `PATCH /api/incidents/{incident_id}/acknowledge`

**Auth**: JWT bearer — roles `admin` or `noc_agent` (`403` for `analyst`)

Body: `{ "acknowledged_at": "2026-07-07T09:20:00Z" }` (optional, defaults to now).

Sets `acknowledged_at` and flips status `open → acknowledged` (no-op if
already acknowledged/resolved/closed). Invalidates the `kpi:alerts*` cache
prefix. `404` if the incident doesn't exist.

## Auth endpoints

All under `/api/auth`.

### `POST /api/auth/login`

Body: `{ "username": "admin", "password": "admin123" }`

Returns `{ access_token, token_type: "bearer", user: {id, username, full_name, role} }`.
`401` with `"Identifiants invalides"` on bad credentials or inactive user.

### `POST /api/auth/pin-login`

Body: `{ "pin": "1234" }` (4–8 digits).

Same response shape as `/login`. Looked up by SHA-256 hash of the PIN
(`dim_user.pin_hash`), not by iterating users — see
[database-schema.md](database-schema.md) for why. `401` with `"Code PIN invalide"`
on no match.

### `GET /api/auth/me`

**Auth**: JWT bearer. Returns the current user (session-restore on frontend
reload). `401` if the token is missing/invalid/expired or the user is inactive.

## Report endpoint

### `GET /api/report/monthly`

**Auth**: JWT bearer (any role) **or** the static API key — the latter lets
the scheduled ETL job (`etl.generate_monthly_report`, 1st of month 02:30)
archive reports without a user session.

Query: `month`, `year`, `format` (`json` default, `pdf`, or `docx`)

- `format=json`: structured report data (same underlying KPI/SLA/incident
  numbers used elsewhere, assembled by `report_service.build_report_data`).
- `format=pdf`: renders the same data via `fpdf2` and returns
  `application/pdf`.
- `format=docx`: renders via `python-docx` and returns a Word document
  (`application/vnd.openxmlformats-officedocument.wordprocessingml.document`).

Both binary formats set `Content-Disposition: attachment;
filename=rapport-noc-{year}-{month:02d}.{format}`. The dashboard exposes them
from the download button in the header (current period, PDF or DOCX).

## WebSocket: real-time alerts

### `WS /ws/alerts?token=<JWT>`

Pushes every newly ingested incident to connected dashboards
(`backend/app/routes/ws.py`). The JWT goes in the **query string** — browsers
can't set an `Authorization` header on a WebSocket handshake. An invalid or
missing token closes the connection with code `4401`.

Messages are JSON, one per frame:

```json
{
  "type": "incident",
  "incident_id": 5821,
  "node_code": "OUA-012",
  "node_name": "…",
  "severity": "critical",
  "status": "open",
  "detected_at": "2026-07-07 09:14:00",
  "description": "Batterie onduleur déchargée"
}
```

A `{"type": "ping"}` heartbeat is sent after ~20s of silence so idle proxies
don't drop the connection. Transport: ingest publishes to the Redis channel
`noc:alerts` (`alert_broadcaster.py`); each WebSocket connection holds its own
subscription, so it works across multiple Uvicorn workers. NGINX proxies
`/ws/` with the required `Upgrade`/`Connection` headers.

The frontend consumer is `frontend/src/hooks/useRealtime.js`: on each
`incident` frame it invalidates the react-query `alerts`/`kpi` caches;
15-second polling remains as a fallback if the socket can't connect.

## Error responses

FastAPI's default error shape: `{ "detail": "<message>" }`.

| Status | When |
|---|---|
| `401` | Missing/invalid API key or JWT; bad login credentials; expired token |
| `403` | Valid JWT but insufficient role (e.g. `analyst` calling `resolve`/`acknowledge`) |
| `404` | Unknown `node_code` on ingest; unknown `incident_id`; unknown `locality_id` |
| `422` | Query/body validation failure (e.g. `month` out of `1..12`, bad `source_tool` enum) |
| `429` | Rate limit exceeded (see [Rate limiting](#rate-limiting)); includes `Retry-After: 60` |
